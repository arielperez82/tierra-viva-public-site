# Tierra Viva Website Visual Redesign -- Roadmap and Delivery Plan

## Executive Summary

The Tierra Viva public website will undergo a visual redesign to evolve from its current functional but utilitarian presentation into an "Institutional Elegance" aesthetic befitting a serious investment platform. The work is scoped entirely to CSS, typography, layout, and new Astro components -- no backend, no database, no authentication. The current site (Astro 5 + Tailwind CSS v4, deployed to GitHub Pages via CI/CD) is the foundation. Seven phases move from design tokens and typography through hero redesign, a new StatsStrip component, homepage section upgrades, a motion/animation system, navigation and footer polish, and optional inner-page refinement. Each phase produces a shippable site. The entire effort is estimated at 5-8 working days for a single developer following TDD.

### Phase 5: Component System and Motion

**Goal:** Add a cohesive animation layer across the site that creates a sense of polish and intentionality. All animations are progressive enhancements that degrade gracefully.

| Attribute              | Detail                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Key Deliverables**   | 1. **Scroll-reveal system**: Reusable CSS class (e.g., `.reveal`) paired with a single IntersectionObserver script. Elements fade-in and translate-up on scroll. Applied to section headings and card groups across all homepage sections. 2. **Card hover elevation**: Utility class (e.g., `.card-elevate`) that adds `transition: transform, box-shadow` on hover. Applied to pillar cards, research cards, partner cards. 3. **Header scroll behavior**: Header transitions from transparent background (when overlapping the hero) to solid white with shadow on scroll. Uses a scroll listener or IntersectionObserver on the hero sentinel. 4. **Link underline animation**: Underlines on internal links animate width on hover (from 0% to 100%). 5. **Reduced-motion override**: All of the above respect `prefers-reduced-motion: reduce` by disabling transitions, transforms, and opacity animations. |
| **Files Touched**      | `src/styles/global.css` (animation utilities, reduced-motion rules), `src/components/Header.astro` (scroll behavior script), new `src/scripts/reveal.ts` (IntersectionObserver), homepage section components (add `.reveal` classes).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| **Dependencies**       | Phase 1 (typography/colors). Phase 2 (hero must exist for header transparency behavior). Phase 4 recommended (cards must exist to receive elevation). Can begin after Phase 2; card hover can be applied iteratively as Phase 4 sections land.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     |
| **Definition of Done** | Vitest tests confirm reduced-motion media query exists in global.css output. Scroll-reveal script is under 1KB minified. Header transitions are smooth (no layout shift -- use `will-change: background-color` sparingly). All animations disabled when `prefers-reduced-motion: reduce` is active (testable via emulation). Lighthouse performance score does not regress. No cumulative layout shift (CLS) introduced by animations.                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| **Risk Flags**         | Transparent header over hero requires careful z-index management and contrast testing (text must remain readable over both cream hero and white header states). Scroll listeners can cause jank if not throttled -- prefer IntersectionObserver over `scroll` events. The reveal system must not flash unstyled content (elements should be visible by default, enhanced with animation only when JS loads).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Estimated Effort**   | 1 day                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |

## Delivery Milestones

The site is deployed continuously to GitHub Pages via GitHub Actions. Each milestone represents a point where the site is in a shippable, visually coherent state.

| Milestone                   | After Phase  | What Ships                                            | Shippable?     |
| --------------------------- | ------------ | ----------------------------------------------------- | -------------- |
| **M0: Baseline**            | -- (current) | Current site as-is on `main`                          | Yes (live now) |
| **M1: New Type System**     | Phase 1      | Upgraded typography and colors, same layout           | Yes            |
| **M2: Hero Impact**         | Phases 2 + 3 | Redesigned hero + stats strip; rest of page unchanged | Yes            |
| **M3: Full Homepage**       | Phase 4      | All homepage sections upgraded                        | Yes            |
| **M4: Polished Experience** | Phase 5      | Animations, scroll reveal, header behavior            | Yes            |
| **M5: Complete Redesign**   | Phase 6      | Footer redesign, full homepage experience             | Yes            |
| **M6: Site-Wide Polish**    | Phase 7      | Inner pages refined (stretch)                         | Yes            |

**Recommended ship points:** M1, M2, and M5 are the highest-value checkpoints. M1 can ship to production immediately as it is non-breaking. M2 is the first moment the redesign is visibly apparent to stakeholders. M5 is the "complete" milestone for the homepage experience.

## Critical Path

```text
Phase 1 (Foundation)
  |
  +---> Phase 2 (Hero) ---> Phase 5 (Motion, partial -- header transparency)
  |       |
  |       +---> Phase 4 (Section Upgrades) ---> Phase 5 (Motion, card hover + reveal)
  |
  +---> Phase 3 (Stats Strip) --- can run parallel with Phase 2
  |
  +---> Phase 6 (Footer) --- can run after Phase 1, independent of 2-5
  |
  +---> Phase 7 (Inner Pages) --- can run after Phase 1 + Phase 6
```

**Sequential dependencies (critical path):**

1. Phase 1 blocks everything. Start here.
2. Phase 2 blocks Phase 5 header behavior.
3. Phase 4 is recommended before Phase 5 card animations (but not strictly required).

**Parallelizable work:**

- Phase 2 and Phase 3 can run in parallel after Phase 1.
- Phase 6 can run in parallel with Phases 2-5 (only depends on Phase 1).
- Phase 4 sub-tasks (4a through 4f) are independent and can be parallelized.
- Phase 7 can start as soon as Phases 1 and 6 are done.

**Shortest path to "complete homepage redesign" (M5):**
Phase 1 -> Phase 2 + Phase 3 (parallel) -> Phase 4 -> Phase 5 -> Phase 6
